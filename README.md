# Проектирование высоконагруженного сервиса стримингового кинотеатра - Кинопоиск
## Часть 1. Тема, функционал и аудитория

### Тема
Тема данной курсовой работы - проектирование высоконагруженного сервиса стимингового кинотеатра на примере [Кинопоиска](https://www.kinopoisk.ru/)
### Целевая аудитория
- Общее число пользователей - 60 млн [^1]
    - из них 18 млн - смотрящих по мультиподписке Яндекс Плюс в месяц  [^1]
    - Один подписчик проводит в среднем 16 дней в месяц пользуюясь Кинопоиском [^1]
    DAU = 18 млн * 16 / 30 =  9.6 млн

### Ключевой функционал 
- Просмотр каталога фильмов, сериалов и телеканалов
- Поиск по каталогу
- Возможность добавить фильм или сериал в Избранное (“Буду смотреть”)
- Воспроизведение видео
- Регистрация и авторизация пользователей
- Возможность оставить оценку, отзыв, рецензию на фильм, сериал

### Ключевые продуктовые решения
- Доступ к кинотеатру по мультиподписке Яндекс Плюс
- Кроссплатформенный доступ к фильмам, сериалам и телеканалам
- Система, анализирующая более 800 маркеров поведения подписчика для формирования рекомендаций [^2]
- Сквозная синхронизация прогресса и списков между устройствами
- Тяжелый видеоконтент выносится в CDN [^3]

## Размер хранения в разбивке по типам данных

### Пользовательские данные (БД)

Берём MAU платформы = `60 000 000` пользователей [^1].

Формула:
- `Vuser_db = MAU_platform * Suser`

Расчёт:
- `Vuser_db = 60 000 000 * 24 КБ = 1 440 000 000 КБ ≈ 1.44 ТБ`


### Каталог, поиск и медиа

#### Каталог и поисковый индекс
Известно:
- библиотека онлайн-кинотеатра: около **80 тыс. единиц контента** [^6]

Оценка метаданных:
- метаданные + связи + описания: `~10 КБ / единицу контента`
- поисковый индекс: `~2–3x` от объёма метаданных

| Тип данных | Количество | Оценка | Итог |
|---|---:|---:|---:|
| Метаданные каталога | 80 000 | ~10 КБ / единицу | ~0.8 ГБ |
| Поисковый индекс | 80 000 | ~2–3x от метаданных | ~2 ГБ |
| Постеры/изображения (основные копии) | 80 000 | ~0.5–1 МБ | ~40–80 ГБ |

#### Видеоконтент (порядок величины)

Для оценки объёма видеобиблиотеки используем диапазон средней длительности одной единицы контента:
- `45–90 минут` (единица контента включает фильмы, серии и другие типы контента)
- суммарный битрейт набора ABR-профилей = `~12.1 Мбит/с` [^5]

Размер одной единицы контента:
- при 45 мин:
  - `Sunit_video_45 = 12.1 Мбит/с * 2700 с / 8 ≈ 4.09 ГБ`
- при 90 мин:
  - `Sunit_video_90 = 12.1 Мбит/с * 5400 с / 8 ≈ 8.17 ГБ`

Общий объём видеобиблиотеки:
- при 45 мин: `80 000 * 4.09 ГБ ≈ 327 ТБ`
- при 90 мин: `80 000 * 8.17 ГБ ≈ 654 ТБ`

Итого порядок объёма видеоконтента:
- **`~327–654 ТБ`**


## Сетевой трафик

Разделяем трафик на:
1. **API / управляющий** (каталог, поиск, избранное, auth, прогресс)
2. **Видеотрафик (CDN)**

### Параметры для видеотрафика

| Параметр | Значение | Комментарий |
|---|---:|---|
| Средняя длительность просмотра в день `Twatch` | 60 мин | Оценка для активного смотрящего пользователя |
| Средний битрейт `Bvideo` | 3 Мбит/с | Усреднение по качествам |
| Длительность сегмента `Tseg` | 6 сек | Типичный порядок для HLS/DASH [^4] |
| Коэффициент пиковости `Kpeak` | 2 | Пик/среднее по суткам |


### Видеотрафик (CDN)

#### Средняя одновременность
- `Twatch = 60 мин = 3600 сек`
- `Cavg = DAU * Twatch / 86400`

Расчёт:
- `Cavg = 9 600 000 * 3600 / 86400 = 400 000`

#### Пиковая одновременность
- `Cpeak = Cavg * Kpeak = 400 000 * 2 = 800 000`

#### Средняя и пиковая полоса видеотрафика
- `BWavg_video = Cavg * Bvideo`
- `BWpeak_video = Cpeak * Bvideo`

Расчёт:
- `BWavg_video = 400 000 * 3 Мбит/с = 1200 Гбит/с`
- `BWpeak_video = 800 000 * 3 Мбит/с = 2400 Гбит/с`

#### Суммарный суточный видеотрафик
Формула:
- `Traffic_day_video = DAU * Twatch * Bvideo`

Расчёт:
- `Traffic_day_video ≈ 9.6 млн * 60 мин * 3 Мбит/с ≈ 12.96 ПБ/сутки`

> По нагрузке доминирует видеотрафик, поэтому архитектурно он выносится в CDN [^3].


## RPS в разбивке по типам запросов (средний и пиковый)

### Формулы расчёта
- `RPD = DAU * Naction`
- `RPSavg = RPD / 86400`
- `RPSpeak = RPSavg * Kpeak`, где `Kpeak = 2`

### RPS для backend API (без видео-сегментов)

| Тип запроса | Действий на пользователя в день | RPD (запросов/сутки) | Средний RPS | Пиковый RPS |
|---|---:|---:|---:|---:|
| Просмотр каталога | 20 | 192 000 000 | 2 222.22 | 4 444.44 |
| Поиск по каталогу | 2 | 19 200 000 | 222.22 | 444.44 |
| Избранное (add/remove) | 0.2 | 1 920 000 | 22.22 | 44.44 |
| Старт воспроизведения | 1 | 9 600 000 | 111.11 | 222.22 |
| Авторизация / refresh | 0.1 | 960 000 | 11.11 | 22.22 |
| Оценка / отзыв / рецензия | 0.03 | 288 000 | 3.33 | 6.67 |
| Heartbeat прогресса | 120 | 1 152 000 000 | 13 333.33 | 26 666.67 |
| **Итого API (без CDN)** | — | **1 375 968 000** | **15 925.56** | **31 851.11** |

- **Суммарный RPSavg API = 15 926**
- **Суммарный RPSpeak API = 31 851**

### RPS для видео-сегментов (CDN)

При сегменте `6 сек` [^4]:
- один зритель запрашивает примерно `1/6` сегмента в секунду

Формулы:
- `RPSavg_segments = Cavg / 6`
- `RPSpeak_segments = Cpeak / 6`

Расчёт:
- `RPSavg_segments = 400 000 / 6 ≈ 66 666.67`
- `RPSpeak_segments = 800 000 / 6 ≈ 133 333.33`

| Тип запроса | Средний RPS | Пиковый RPS |
|---|---:|---:|
| Запросы видео-сегментов (CDN) | 66 666.67 | 133 333.33 |


## Сводная таблица трафика (пик / суточный)

### API / управляющий трафик (без видео)

Для API-трафика используем упрощённую оценку средней “стоимости” запроса по трафику:
- **средний обмен на 1 API-запрос ≈ 8 КБ** (с учётом смеси типов запросов)

Тогда:
- `Traffic_day_api = 1 375 968 000 * 8 КБ ≈ 11.01 ТБ/сутки`
- `BWavg_api ≈ 1.02 Гбит/с`
- `BWpeak_api ≈ 2.04 Гбит/с` (при `Kpeak = 2`)

| Тип трафика | Средняя полоса | Пиковая полоса | Суточный трафик |
|---|---:|---:|---:|
| API / управляющий (без видео) | ~1.02 Гбит/с | ~2.04 Гбит/с | ~11.01 ТБ/сутки |

### Видеотрафик (CDN)

| Тип трафика | Средняя полоса | Пиковая полоса | Суточный трафик |
